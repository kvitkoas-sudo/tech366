Подсистема почтовых рассылок

Хранение получателей, формирование писем и журналирование результата отправки в конфигурации товародвижения.

Источник: СтруктураХраненияБД_обогащенная.xml Версия документа: 1.0 Дата: 21.07.2026
Аналитики Разработчики 1С
О документе Описаны только объекты, фактически присутствующие в структуре хранения БД конфигурации. Прикладной код (модули отправки, регламентные задания) в структуру хранения не входит и здесь описывается по назначению объектов, а не по исходному коду. Метки АналитикРазработчик у заголовков указывают основную аудиторию раздела.

01Назначение

Подсистема решает три задачи: хранит списки получателей (адреса, аптеки, контрагенты), хранит подготовленные к отправке письма с темой и текстом и фиксирует результат отправки — статус и текст ошибки. Это штатный механизм конфигурации, который можно переиспользовать для новых сценариев оповещения без создания собственных объектов.

Ключевая особенность: получатель может задаваться не только строкой-адресом, но и ссылкой на аптеку (Справочник.МестаХранения) или на контрагента (Справочник.Поставщики) — то есть адрес разрешается из карточки объекта, а не вводится вручную.

02Состав и архитектура

Подсистема логически делится на четыре слоя: получатели, очередь/журнал писем, классификаторы (виды и статусы) и транспорт (настройки SMTP).

03Жизненный цикл письма

Стандартный поток: сценарий формирует запись в регистре со статусом «подготовлено», фоновый механизм отправляет её через транспорт SMTP и проставляет итоговый статус. Проведение исходных документов при этом не блокируется.

01
Формирование
Сценарий создаёт запись в ПочтовыеРассылки: вид, получатели, тема, текст.
→
02
Подготовлено
Статус ПодготовленоКОтправке — письмо ждёт в очереди.
→
03
Отправка
Фоновый механизм берёт настройки SMTP и отправляет письмо пообъектно.
→
04
Результат
Отправлено либо ОшибкаОтправки + текст в ОписаниеОшибки.
Аналитику Регистр играет двойную роль — очередь (пока статус «подготовлено») и журнал (после отправки хранит результат). Это то, что в требованиях к оповещениям обычно называют «журналом отправок»: адресат, тема, результат, ошибка. Отдельно создавать журнал не требуется.

04Справочник «Рассылки»АналитикРазработчик

Хранит именованные рассылки. Сам справочник содержит только стандартные поля (код, наименование); вся смысловая нагрузка — в трёх табличных частях получателей. Три способа адресации позволяют собрать один список из произвольных адресов, аптек и контрагентов.

05Регистр «ПочтовыеРассылки»АналитикРазработчик

Непериодический независимый регистр сведений. Хранит подготовленные и отправленные письма — это одновременно очередь отправки и журнал результата.

ВидРассылкиПеречислениеСсылка.ВидыПочтовыхРассылокКлассификация письма по сценарию
ПолучателиСтрока(150)Список адресов строкой (не ссылка!)
ТемаПисьмаСтрока(150)Тема

ОписаниеОшибкиСтрока(500)Текст ошибки при неудаче
Ограничение Все поля регистра — ресурсы/атрибуты, измерений нет. Уникального ключа по получателю или документу не существует, поэтому записи только добавляются (журнал), а не перезаписываются по ключу. Ссылки на документ-основание (например, документ поступления) в регистре нет — если требуется связь «письмо ↔ документ», её нужно добавлять отдельно (см. раздел 11).

Постановка письма в очередь (создание записи регистра)

Запись = РегистрыСведений.ПочтовыеРассылки.СоздатьМенеджерЗаписи();
Запись.ВидРассылки    = Перечисления.ВидыПочтовыхРассылок.РаспоряжениеНаРеализацию;
Запись.Получатели     = СтрокаАдресов;          // адреса через ";"
Запись.ТемаПисьма     = Тема;
Запись.ТекстПисьма    = Текст;
Запись.СтатусОтправки = Перечисления.СтатусыОтправкиПисьма.ПодготовленоКОтправке;
Запись.Записать();
Замечание Регистр без измерений — при Записать() запись добавляется, а не заменяется по ключу. Для отметки результата после отправки создаётся/дописывается запись с итоговым статусом согласно принятой в сценарии логике.

10Расширение под новый сценарийРазработчик

Чтобы задействовать подсистему для нового типа оповещения (например, оповещение аптек о сторно МОР), минимально необходимо:

Шаг 1
Новый вид
Добавить значение в Перечисление.ВидыПочтовыхРассылок.
→
Шаг 2
Формирование
В точке-триггере собрать адреса и текст, записать в ПочтовыеРассылки со статусом «подготовлено».
→
Шаг 3
Отправка
Отдать письмо штатному фоновому механизму отправки; проставить итоговый статус и ошибку.
Проектное решение Если сценарию нужна жёсткая связь «письмо ↔ документ-основание», штатный регистр её не хранит (поле Получатели — строка, ссылки на документ нет). Тогда решается на уровне ТЗ: либо смириться с адресацией строкой и без ссылки на документ, либо завести отдельный регистр с измерением-документом по образцу ОчередьСоздатьОтправитьУПДВСбис.
